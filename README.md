## New host bootstrap (sops)

Secrets in `sops/secrets.yaml` are encrypted to age keys. Each host's
key is derived from its SSH host key (`sops.age.sshKeyPaths` in
`base.nix` — `/etc/ssh/ssh_host_ed25519_key`). A fresh machine can't
decrypt anything (no wifi profiles, no user password) until it is added
to the recipient list:

1. Get the new host's SSH host key. Normally generated by the installer
   on first boot; for remote installs (nixos-anywhere) pre-generate one
   and copy it into place so the first rebuild already has it:
   `ssh-keygen -t ed25519 -N "" -f ssh_host_ed25519_key`
2. Convert the **public** key to an age key:
   `nix run nixpkgs#ssh-to-age -- -i ssh_host_ed25519_key.pub`
3. Add it to `sops/.sops.yaml`: a `&age_<hostname>` anchor under `keys:`
   and a reference under `creation_rules:`.
4. Re-encrypt the secrets to include the new recipient — run from a
   machine whose key is already in the recipient list:
   `sops updatekeys sops/secrets.yaml`
5. Add the per-host secrets the config expects (e.g.
   `user-password-<hostname>`: `nix run nixpkgs#mkpasswd -- -m yescrypt`,
   then `sops sops/secrets.yaml`).
6. Rebuild on the new host.

Notes:
- Editing `secrets.yaml` (steps 4–5) requires holding a private key
  matching one of the `creation_rules` recipients — e.g. the user-level
  age key in `~/.config/sops/age/keys.txt`, or a listed host's SSH key.
- If a host is reinstalled and gets a new host key, repeat steps 2–4
  with the new key and drop the old anchor.
- Under impermanence the host key must live on the persistent volume
  (`/persist/etc/ssh/...`) and `sops.age.sshKeyPaths` updated to match —
  losing it means re-bootstrapping the host into sops.
